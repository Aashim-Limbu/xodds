# Settlement verifies TxLINE's Merkle proof in our own program, permissionlessly

Our settlement instruction takes TxLINE's on-chain score root (`daily_scores_roots` PDA) as a read-only account plus the Fixture's score proof, and **verifies the Merkle inclusion proof itself, on-chain**, before paying out. The pot is released only if the proof checks out against TxLINE's published root. Anyone can submit a settlement — it is **permissionless**; a Keeper bot we run does it automatically at the final whistle for UX, but a dead Keeper cannot strand a Pool.

We rejected two alternatives. (1) **CPI into TxLINE's `validateStat`** — the TxLINE docs only ever invoke it off-chain via `.view()`; CPI/return_data is undocumented and the program name/IDL could not be confirmed, so building on it blind is a demo-day risk. (2) **Trusted off-chain resolver** — our backend runs `.view()` and instructs payout — is simplest but makes our server the party that decides the winner, which destroys the app's entire differentiator (that it is trustless). Re-implementing the ~32-byte Merkle hash-walk on-chain is more Rust but is the only path where *nobody, including us, can choose the outcome*. Fallback to the trusted resolver only if in-program verification exceeds the compute budget (the integration workflow indicates it fits within 1.4M CU).
